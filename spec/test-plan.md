## 実装優先度

| 優先度 | テスト | 理由 |
|--------|--------|------|
| ★★★ | W-A（条件進捗チェックマーク） | 直近で実装したばかり・デグレしやすい |
| ★★★ | W-B（報酬受取ボタン） | 直近で実装。満杯対応含め動作保証が必要 |
| ★★★ | MC-A（報酬dropItem） | 直近修正。満杯時のフォールバック未検証 |
| ★★★ | MC-C（部分進捗API） | condition-progressテストに自然に追加できる |
| ★★ | MC-D（SSE部分達成） | 実装はあるがmc-testsで未検証 |
| ★★ | W-C（読み取り専用モーダル） | 何度も再発しているバグ種別 |
| ★★ | W-G（URLログイン） | 仕様としてあるが未テスト |
| ★ | W-D（トースト・ツールバー） | 目視確認で済む範囲だが自動化する価値あり |
| ★ | W-E（エッジ保存） | 基本動作だが保存永続化未確認 |
| ★ | W-F（パン初期位置） | UI品質。ピクセル判定なので脆い |
| ★ | W-H（スマホ進捗・報酬） | スマホはデグレしやすい |
| ★ | MC-B（advancement） | RCON権限依存で環境依存が大きい |

## 実装方針

### web/tests への追加方法

- 既存の `editor.spec.ts` にテストブロックを追記する（ファイルを増やしすぎない）
- モックサーバー（`web/mock/server.ts`）の `/api/progress` と `/api/progress/{id}/claim` のレスポンスを制御してテスト
- SSE テストは既存の `20系` と同じパターン（`page.evaluate` でカスタムイベントを発火）で行う

### mc-tests への追加方法

- `condition-progress.test.js` に `MC-C` を追記
- `quest-notification.test.js` に `MC-D` を追記
- `reward-claim.test.js` を新規作成（`MC-A`）
- `advancement-progress.test.js` を新規作成（`MC-B`）— 環境依存が高いので最後

### モック固定データの前提

- クエスト ID=1: advancement 条件のみ
- クエスト ID=2: item 条件 × 2（stone×1・dark_oak_log×5）
- クエスト ID=3: checkmark 条件（手動確認）
- クエスト ID=4: 報酬あり（item: stone×1）

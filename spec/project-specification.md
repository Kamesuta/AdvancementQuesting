# AdvancementQuesting プロジェクト仕様書

## プロジェクト概要

FTB Quest・Better Questing・HQM などのクエスト MOD と同様の機能を持つ Minecraft プラグイン。
プレイヤーがクエストを受注し、達成条件をクリアして報酬を得られるシステムを提供する。

### 特徴

- ブラウザで動くマップベースのクエストエディタ（PC・スマホ対応）
- 編集権限を持たないプレイヤーによるクエスト提案・投票システム
- Minecraft の Advancement システムとの統合
- クエストデータは JSON ファイルで管理（Git 差分が読める）

---

## システム構成

```
┌──────────────────────────────────────────────────────────┐
│  ブラウザ (React SPA)                                      │
│  クエストマップエディタ / 進捗確認 / 提案・投票             │
└────────────────────┬─────────────────────────────────────┘
                     │ REST API  /api/*
         ┌───────────┴───────────┐
         │                       │
┌────────▼────────┐    ┌─────────▼────────────┐
│   Mock Server   │    │  Minecraft Plugin     │
│ Express+SQLite  │    │  Paper / Java         │
│ (開発・テスト)   │    │  組み込み HTTP サーバー │
└─────────────────┘    └──────────────────────┘
                                 │
                       ┌─────────┴────────────┐
                       │                      │
                  JSON ファイル           SQLite DB
                  (クエスト定義)          (進捗・セッション)
```

### 本番構成

- ビルド済み静的ファイル (`dist/`) をプラグインの組み込み HTTP サーバーから配信
- API もプラグイン内で実装 — 追加サーバー不要

### 開発構成

- フロントエンド: Vite dev server (デフォルト `localhost:5173`、E2E テスト時は `localhost:5174`)
- API: Mock サーバー (デフォルト `localhost:3000`、E2E テスト時は `localhost:3001`)
- Vite proxy で `/api/*` を Mock サーバーへ転送

## 機能一覧

### A. 認証

#### A-1. ログイン方法

**6桁コード認証（メイン）**
```
/quest code
→ 6桁の認証コードを生成 (有効期限: 5分)
→ Web ページでコードを入力
→ セッション確立、ロールをトークンに紐付け
```

**URL リンク認証（補助・将来対応）**
```
/quest login
→ プレイヤー固有の URL を生成
→ チャットにクリック可能なリンクを表示
→ URL を開くと自動でセッション確立
```

**クイックログイン（開発・Mock 専用）**
```
POST /api/auth/quick
→ 固定の開発用トークンでセッションを即座に作成/更新
→ 本番プラグインには実装しない
```

#### A-2. 権限ロール

| ロール | 取得条件 | 主な操作 |
|--------|---------|---------|
| `player` | ゲームにログインしている全員 | クエスト閲覧・進捗確認・提案投稿・投票 |
| `editor` | OP または権限ノード `aq.editor` | クエスト作成・編集・提案の承認/却下 |
| `admin` | OP | editor の全権 + クエスト削除 |

ロールはセッション発行時にプラグインが決定し、Web 側は再判定しない。

#### A-3. セッション管理

- ログアウトはセッションを**ハード削除**する（ソフトデリート不可）
- 同じトークンで再ログインする場合は upsert で有効期限を更新する
- フロントエンドは `localStorage` にトークンを保持し、`Authorization: Bearer <token>` で送る

### B. クエストエディタ（Web）

マップベースのビジュアルエディタ。`editor` / `admin` ロールのみ編集操作可能。

#### B-1. クエストデータ構造

実装上のフィールド（Mock スキーマ準拠）:

```
クエスト:
  - id          (UUID, 自動生成)
  - title       (タイトル)
  - description (説明文)
  - icon        (Minecraft アイテム ID)
  - category    (カテゴリ文字列)
  - prerequisites (前提クエスト ID 配列)
  - conditions  (達成条件 JSON 配列)
  - rewards     (報酬 JSON 配列)
  - mapPosition ({x, y} または null)
  - customButtons (カスタムボタン JSON 配列)
  - status      (draft / proposed / public / hidden)
  - creatorUuid
  - createdAt / updatedAt
```

> **注意:** 仕様上は `subtitle`・`tags`・`tasks`・`rewardTableId` が設計されているが、
> 現 Mock は `description`・`category`・`conditions`・`rewards` として実装されている。
> プラグイン実装時に最終フィールド名を確定する。

#### B-2. 条件（conditions）種別

| タイプ | 説明 | 必要パラメータ |
|--------|------|-------------|
| `item` | アイテムを集める | `itemType`, `count` |
| `advancement` | Advancement を達成する | `advancementId` |
| `checkmark` | 管理者が手動でチェック | `label` |
| `command` | コマンド実行で達成 | `command` |

#### B-3. 報酬（rewards）種別

| タイプ | 説明 | 必要パラメータ |
|--------|------|-------------|
| `item` | アイテム付与 | `itemType`, `count` |
| `experience` | 経験値付与 | `amount` |
| `command` | コマンド実行 | `command` |
| `permission` | 権限ノード付与 | `node` |
| `money` | お金付与 (Vault) | `amount` |

#### B-4. マップエディタ操作

| ツール | 操作 |
|--------|-----|
| 選択・移動 | ドラッグでノード移動、背景ドラッグでパン |
| クエスト追加 | add_quest モードでクリック |
| 依存関係 | connect_quest モードでノード→ノードをクリック |
| クエスト編集 | edit_quest モードでノードをクリック→モーダル |
| 削除 | delete_quest モードでノードをクリック |

#### B-5. 保存

- 「💾 保存」ボタンで `PUT /api/quests/:id`（全クエスト一括） + マップ座標はクエスト内の `mapPosition` に埋め込んで保存
- 保存後にキャッシュ無効化して再取得

### C. クエスト提案システム ✅ 実装済み

編集権限を持たないプレイヤーが、クエストの追加を申請する仕組み。

#### フロー

```
1. player が「✨ クエスト追加を提案する」ボタンで提案モード開始
2. 通常エディタと同じ UI でクエスト内容・マップ位置を設定（下書き）
3. 「📤 提案を送信する」でサーバーへ保存
   → 提案モードは継続（提案済みクエストをマップ上で確認可能）
4. 「✕ 終了」で提案モード終了 → 提案クエストは非表示
5. 再度提案モードを有効にすると自分の提案クエストが表示される
6. editor / admin が提案一覧サイドパネルで確認・承認/却下
7. 承認時: クエストを `public` に変更 → 全員に即時表示
   却下時: `hidden` に変更、却下理由を記録
```

#### 提案の制約

- マップ座標を含む（`mapPosition` としてクエストに埋め込み）
- 提案内容は `quests` テーブルに `status='proposed'` として作成し、`quest_proposals` から参照
- 1プレイヤー1票（反対方向で投票すると上書き、同方向で再投票すると取り消し）
- 提案モード中は既存クエストノードの編集モーダルを開けない（提案ドラフトのみ編集可）
